Chapter 7

Numeric Path and Optimizer Hygiene

The numeric path is where many “boring details” live: precision choice, reduction dtype, attention softmax dtype, optimizer-state storage, clipping, and checkpoint recovery. Public large-scale evidence argues that these details are not secondary. They are part of the stability stack, especially when the model is already architecturally reasonable. PaLM OLMo card GLM-130B

Why This Chapter Exists

Architecture-first explanations are necessary but incomplete. A run with sensible residual geometry can still die because FP16 softmax or reduction paths are fragile, because optimizer state is stored too aggressively in low precision, or because the training system has no recovery policy when a rare spike still occurs. This chapter exists to make those engineering decisions explicit.

Intuition

The numeric path sits between the theory of the update and the actual arithmetic performed on hardware. If the arithmetic path is too fragile, good architectural choices can be wasted. If the arithmetic path is solid, it will not fix a fundamentally bad architecture, but it will stop easy-to-avoid numeric failures from masquerading as deeper research questions.

Formal Mechanics

The strongest public recommendations in this bundle are:

  • Use BF16 where possible.
  • If forced into FP16, keep sensitive operations in FP32, especially attention softmax, reductions, and optimizer states.
  • Use global clipping at 1.0.
  • Retain a checkpoint recovery path for rare spikes.

OLMo’s public recipe exposes FP32 gradient reduction and FP32 optimizer states. GLM-130B explicitly reports FP32 attention softmax in an FP16 training stack. These are unusually concrete public implementation facts. OLMo card GLM-130B

Example

PaLM is the most instructive operational example. Even with global grad clip = 1.0 and z-loss = 1e-4 · log² Z, the 540B run still reported roughly 20 irregular spikes. The mitigation was not a philosophical change to the model. It was a recovery playbook: restart from about 100 steps before the spike and skip roughly 200–500 nearby data batches. PaLM

Strengths

  • These controls are usually cheap relative to architectural redesign.
  • They are grounded in real public implementation and operations evidence.
  • They reduce wasted compute and make failure forensics more interpretable.

Limitations And Failure Cases

  • Numeric hygiene can mask, but not eliminate, a deeper architectural root cause.
  • Global clipping constrains the update only after the bad gradients already exist.

Practical Notes

If hardware allows it, BF16 is the cleanest default. If not, be explicit about which operations remain FP32. Do not treat that as a hidden implementation detail. Record it in configs and dashboards. Also, a stable training program should define in advance when to roll back, how far to roll back, and whether to skip suspect data batches after a spike. Recovery policy is part of reproducibility.
