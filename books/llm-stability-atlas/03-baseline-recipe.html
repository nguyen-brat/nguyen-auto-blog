Chapter 3

Stability Baseline Recipe

This chapter states the default stack that should already be present before any team spends serious time on exotic stabilizers: Pre-LN or RMSNorm, warmup, global grad clip 1.0, FP32-sensitive operations where needed, explicit [eod] packing, attention-logit control, and z-loss. PaLM OLMo OLMo2 Docs Gemma2 Docs

Why This Chapter Exists

Stability conversations often jump too quickly to specialty methods because frontier ideas are more memorable than baseline discipline. That is backwards. The baseline recipe chapter exists to define which controls should already be present in almost every serious decoder-LM run before any team claims that its remaining problem is exotic. A surprising amount of “novel” stabilization work is just a workaround for skipping one or more baseline elements.

The Default Stack

1. Stable normalization geometry

Start from Pre-LN or a closely related RMSNorm-based decoder baseline. The historical reason is still important: Post-LN has initialization-time gradient pathology near the output layer, while Pre-LN addresses that directly. Xiong 2020

2. Warmup and ordinary optimizer hygiene

Even a stronger architecture does not justify treating warmup as optional by default. The open OLMo recipe exposes a concrete warmup of 5000 steps, peak LR 3e-4, weight decay 0.1, epsilon 1e-5, and global grad clip 1.0. OLMo

3. Global grad clipping, but without mythology

Global grad clip 1.0 is part of the public large-model baseline, not because it is sufficient but because it is cheap, familiar, and often necessary. PaLM is the right corrective here: clipping was present and spikes still occurred, which is exactly why this book refuses to treat clipping as a complete solution. PaLM

4. FP32-sensitive numeric path

Keep gradient reductions, optimizer state, and fragile softmax paths in FP32 where the stack needs it. This belongs in the baseline because many otherwise puzzling failures collapse once precision-sensitive paths are cleaned up.

5. Attention-logit control as part of the default

The rerun bundle strengthened one design choice relative to the older report: Q/K normalization or soft-capping should now be taught as baseline-friendly controls, not as niche additions. OLMo2 exposes Q/K RMSNorm; Gemma 2 exposes attention-logit soft-capping. OLMo2 Docs Gemma2 Docs Gemma2 Blog

6. Output-head control through z-loss

PaLM gives the clearest exact public anchor: z-loss = 10^-4 \log^2 Z. That combination of exact coefficient and production-scale precedent is the main reason this handbook keeps z-loss in the default stack. PaLM

7. Explicit packing boundaries

Sequence packing is not an upstream clerical detail. PaLM’s exact 2048-token training sequences and explicit [eod] boundaries are part of the real baseline because they protect later chapters from data-boundary confusions.

Why Each Item Belongs Before Escalation

The baseline stack is not a checklist assembled from unrelated papers. Each item blocks a common misdiagnosis. Pre-LN or RMSNorm blocks early geometry pathology from masquerading as optimizer weakness. Warmup and clipping prevent ordinary large-update failures from polluting later analysis. Numeric-path conservatism prevents dtype fragility from being mistaken for architecture failure. Attention-logit control prevents late sharpening from being misread as generic spike behavior. z-loss prevents output-head drift from being handled through the wrong control family. Explicit packing boundaries prevent EOS and repetition artifacts from being blamed on the decoder.

What The Baseline Is Not

The baseline is not a promise that the run can never spike. PaLM itself is the public counterexample. The baseline is also not the deepest architecture recipe available. It does not assume every stack needs DeepNorm. It does not assume every output-head issue needs OEC. It does not assume every rare spike justifies a spike-aware optimizer. The baseline is therefore best understood as the point after which remaining failures become specific enough to deserve specific chapters.

Escalation Boundary

Once the baseline is in place, escalation becomes conditional. If the run still diverges because depth-driven residual amplification remains, go to Chapter 6. If late spikes align with entropy collapse, go to Chapter 7. If the main abnormality is output-logit drift, go to Chapter 8. If the stack still produces infs or restart-sensitive failures, go to Chapter 5. If repetition persists under sane decoding and clean packing, go to Chapter 9. That ordering is the real payload of the baseline chapter: it defines what “baseline-complete” means.

Strengths And Limitations

Strength. The stack is anchored in public flagship or open-model evidence rather than being inferred from private folklore. It is therefore teachable and auditable.

Limitation. Not every item is equally documented across all modern stacks. OLMo2’s public docs expose many architecture facts but not every appendix-level optimizer constant, which is why some exact modern recipe details remain [MISSING FROM SOURCE].

Practical Notes

If a team wants one blunt operational rule from this chapter, it is this: do not spend a week evaluating frontier stabilizers while the run still lacks one or more baseline controls. The observed failure may be real, but the experimental order is still wrong.
