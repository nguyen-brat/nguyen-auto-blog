Chapter 4

Input Pipeline and Packing

This chapter isolates the upstream side of stability: document boundaries, token streams, fixed-length blocks [B, T], and the way packing mistakes can later look like optimizer, architecture, or decoding failures. PaLM Holtzman Welleck

Why This Chapter Exists

A surprising number of repetition complaints and EOS pathologies begin upstream of the model. If documents are concatenated without explicit boundaries, the model is trained on false continuations. If windowing logic treats boundary tokens inconsistently, the resulting generations can look like decoding bugs or objective-level repetition. This chapter exists because no amount of optimizer cleverness repairs a target distribution that was corrupted during packing.

End-To-End Flow

In the stable pattern taught here, raw documents are tokenized into one long stream, document boundaries are marked with explicit [eod] or EOS-like separators, and the stream is then chunked into fixed-length windows [B, T]. Embedding lookup maps token IDs to hidden states [B, T, D]. Everything later in the network assumes that this boundary discipline is meaningful. PaLM is a strong anchor because it publicly exposes exact-length 2048-token training sequences with explicit end-of-document treatment. PaLM

Packing Failure Signatures

  • Boundary tokens are overproduced or underproduced even when training curves look sane.
  • Generations appear to continue abruptly into semantically unrelated text, as if document seams are learnable continuations.
  • Repetition persists in a way that is suspiciously tied to EOS or document-boundary symbols rather than generic token loops.
  • Objective-level interventions seem necessary earlier than they should, because the model was trained on bad cross-document continuations.

Worked Example

Suppose two unrelated documents are concatenated without an explicit [eod] marker. The model is then trained to predict the opening tokens of the second document as valid continuation targets for the first. At inference time this can surface in at least two misleading forms. First, the model may appear bad at termination because it has learned that hard boundaries are uncommon. Second, the model may look repetitive around seam-like transitions because it has learned noisy continuation rules near document ends. Neither symptom proves a numeric training failure. Both can be generated upstream.

Repair Order

  1. Verify explicit boundary insertion in the raw token stream.
  2. Verify that chunking into [B, T] windows preserves the intended boundary semantics.
  3. Audit EOS handling in both training targets and decoding code.
  4. Only after packing is clean, decide whether repetition remains decode-side or objective-side.

What Packing Cannot Explain

Packing is a powerful upstream cause, but it should not be made to explain everything. It does not cause attention entropy collapse in the sense used by Chapter 7. It does not create output-head drift in the sense used by Chapter 8. It does not explain FP16 softmax brittleness. This negative boundary matters because teams can become so aware of data issues that they start attributing every strange generation artifact to packing. The correct use of this chapter is narrower: use it to rule in or rule out boundary-related errors before moving deeper into model-side or decode-side explanations.

Strengths And Limitations

Strength. Packing hygiene is often the cheapest stability win in the entire system because the repair cost is low and the diagnostic value is high.

Limitation. Clean packing does not solve attention entropy collapse, output-head divergence, or mixed-precision numeric failures. It only prevents data-boundary errors from contaminating those later diagnoses.

Practical Notes

Before adding unlikelihood loss or blaming greedy decoding for every loop, audit boundary handling first. The right order is packing, then decode-side diagnosis, then objective-level intervention. Reversing that order makes the entire repetition discussion noisier.
